# Env Sync

**b** can sync configuration files from upstream git repositories into your project. This is useful for sharing manifests, configuration templates, or any files across repositories while tracking changes and handling conflicts.

## Quick start

```bash
# Discover available profiles from an upstream repo
b env profiles github.com/org/infra

# Add a profile to your b.yaml
b env add github.com/org/infra#monitoring

# Sync everything
b update

# Or sync files manually using SCP-style syntax
b install github.com/org/infra:/manifests/base/** ./base
```

---

## Profiles

Upstream repos can publish named **profiles** — preconfigured file sets that consumers can discover and install without manually writing glob patterns.

### Publishing profiles (upstream repo)

Add a `profiles` section to your `b.yaml` with short, descriptive names:

```yaml
# Upstream repo's b.yaml
profiles:
  base:
    description: "Base Kubernetes manifests for all environments"
    files:
      manifests/base/**:
        dest: base/

  monitoring:
    description: "Prometheus + Grafana stack"
    strategy: merge
    files:
      manifests/monitoring/**:
        dest: monitoring/

  hetzner:
    description: "Hetzner-specific cloud infrastructure"
    files:
      manifests/hetzner/**:
        dest: hetzner/

  # Compose profiles with includes
  staging:
    description: "Staging preset (base + monitoring)"
    includes:
      - base
      - monitoring
    ignore:
      - "**/prod-*"
```

Profiles with `includes` are resolved recursively when installed. The including profile's own fields override included ones. Circular includes are rejected.

### Discovering profiles

```bash
$ b env profiles github.com/org/infra@v2.1

Available profiles from github.com/org/infra @ v2.1:

  base                     Base Kubernetes manifests for all environments
    manifests/base/** → base/

  hetzner                  Hetzner-specific cloud infrastructure
    manifests/hetzner/** → hetzner/

  monitoring               Prometheus + Grafana stack
    manifests/monitoring/** → monitoring/

Install a profile with:
  b env add github.com/org/infra#<name>
```

If the upstream repo has no `b.yaml`, the command falls back to listing the directory structure with suggested install commands.

### Adding a profile

```bash
# Add a profile to your local b.yaml
b env add github.com/org/infra#monitoring

# Pin a specific version
b env add --version v2.0 github.com/org/infra#base
```

This copies the profile's configuration (files, strategy, hooks, etc.) into your local `b.yaml` as an env entry. Profiles with `includes` are automatically flattened. Run `b update` to sync the files.

**Version resolution**: If `--version` is given, the profile is pinned to that version. If omitted, the profile tracks the upstream default branch (HEAD).

### Interactive selection

Browse and select profiles interactively:

```bash
$ b env add -i github.com/org/infra

Available profiles from github.com/org/infra:

  [1] base                 Base Kubernetes manifests
  [2] hetzner              Hetzner-specific infrastructure
  [3] monitoring           Prometheus + Grafana stack
  [4] staging              Staging preset (base + monitoring)

Select profiles (space-separated numbers, e.g. "1 3"): 1 3
Added github.com/org/infra#base to b.yaml (Base Kubernetes manifests)
Added github.com/org/infra#monitoring to b.yaml (Prometheus + Grafana stack)

Run `b update` to sync files.
```

Add multiple profiles at once by entering space-separated numbers. Requires a terminal (not available in CI/CD).

## Configuration

Env entries in your local `b.yaml` define what to sync:

```yaml
envs:
  github.com/org/infra:
    version: v2.0
    strategy: merge
    ignore:
      - "*.md"
    files:
      manifests/base/**:
        dest: base/
      manifests/hetzner/**:
        dest: hetzner/

  # Minimal entry — syncs all files from HEAD
  github.com/org/shared-config:
```

### Fields

| Field | Description | Default |
|-------|-------------|---------|
| `description` | Human-readable description (shown by `b env profiles`) | none |
| `includes` | List of profile names to compose from (profiles only) | none |
| `version` | Git tag, branch, or commit SHA to pin | HEAD |
| `strategy` | How to handle local changes: `replace`, `client`, or `merge` | `replace` |
| `group` | Group name for selective syncing with `--group` | none |
| `ignore` | Glob patterns to exclude from all file matches | none |
| `files` | Map of glob patterns to destination config | all files |
| `onPreSync` | Shell command to run before syncing | none |
| `onPostSync` | Shell command to run after syncing | none |

### Files map

```yaml
files:
  # Bare key — preserve original path structure
  manifests/base/**:

  # String shorthand — set destination directory
  manifests/hetzner/**: hetzner/

  # Full config — dest and per-glob ignore
  manifests/monitoring/**:
    dest: monitoring/
    ignore:
      - "*.test.yaml"
```

### Glob patterns

| Pattern | Matches |
|---------|---------|
| `manifests/base/**` | All files recursively under `manifests/base/` |
| `**/*.yaml` | All `.yaml` files anywhere in the repo |
| `configs/ingress.yaml` | A single literal file |
| `charts/*/values.yaml` | `values.yaml` one directory deep under `charts/` |

### Destination paths

The glob prefix is stripped and the `dest` value is prepended:

| Source file | Glob | Dest | Result |
|------------|------|------|--------|
| `manifests/hetzner/deploy.yaml` | `manifests/hetzner/**` | `hetzner/` | `hetzner/deploy.yaml` |
| `configs/ingress.yaml` | `configs/ingress.yaml` | `config/` | `config/ingress.yaml` |
| `README.md` | `**/*.md` | _(none)_ | `README.md` |

### Labels

Use the same repo multiple times with different configurations:

```yaml
envs:
  github.com/org/infra#base:
    version: v2.0
    files:
      manifests/base/**:
        dest: base/

  github.com/org/infra#monitoring:
    version: v2.0
    strategy: merge
    files:
      manifests/monitoring/**:
        dest: monitoring/
```

Labels create separate lock entries so each group is tracked independently.

## SCP syntax

For one-off syncs without editing `b.yaml`:

```
<repo>[@<version>]:/<glob> [<dest>]
```

```bash
b install github.com/org/infra:/manifests/base/** ./base
b install github.com/org/infra@v2.0:/manifests/hetzner/** ./hetzner
b install --add github.com/org/infra@v2.0:/manifests/** ./config
```

## Merge strategies

When upstream files change and you've also modified the local copy, the **strategy** controls what happens.

### `replace` (default)

Overwrites local files with upstream content. In an interactive terminal, prompts per-file:

```
  system/config.yaml has local changes.
    [r]eplace  [k]eep  [m]erge  [d]iff > _
```

In non-interactive mode (CI/CD), overwrites without prompting.

### `client`

Keeps local files unchanged when modified. Only new files from upstream are written.

### `merge`

Three-way merge using the previous commit as base:

1. **Base** — file at the previously synced commit (from `b.lock`)
2. **Local** — current file on disk
3. **Upstream** — file at the new commit

Conflict markers are inserted when both sides changed the same region:

```
<<<<<<< local
your-value: true
||||||| base
original-value: true
=======
upstream-value: true
>>>>>>> upstream
```

Falls back to replace when the base version is unavailable.

## Managing envs

### Status

Check for upstream changes and local drift without syncing:

```bash
b env status
```

Shows per-env: up-to-date, upstream changes available, locally modified files, missing files, or mode drift.

### Preview

Test what a glob matches before adding it to config:

```bash
b env match github.com/org/infra "manifests/base/**" ./base
```

This populates the git cache but does not modify project files or `b.lock`.

### Remove

Remove an env from `b.yaml` and `b.lock`:

```bash
b env remove github.com/org/infra
b env remove github.com/org/infra#monitoring
b env remove --delete-files github.com/org/infra
```

With `--delete-files`, synced files are also deleted from disk.

### Dry-run

Preview what `b update` would change without writing:

```bash
b update --dry-run
```

### Rollback

Revert envs to their previous commit from `b.lock`:

```bash
b update --rollback
b update --rollback github.com/org/infra
```

## Groups

Tag envs for selective syncing:

```yaml
envs:
  github.com/org/infra#base:
    group: dev
    files:
      manifests/base/**:
        dest: base/

  github.com/org/infra#monitoring:
    group: prod
    files:
      manifests/monitoring/**:
        dest: monitoring/
```

```bash
b update --group=dev
```

## Hooks

Run shell commands before or after syncing:

```yaml
envs:
  github.com/org/infra:
    onPreSync: "echo 'Starting sync...'"
    onPostSync: "kubectl apply -k manifests/"
    files:
      manifests/**:
        dest: manifests/
```

Hooks run in the project root directory and respect `--quiet` mode. They are skipped during `--dry-run`.

## File modes

**b** preserves executable file modes from upstream. Files with git mode `100755` target `0755` permissions; others target `0644`. The initial write respects your system umask; subsequent updates normalize permissions to match upstream. This ensures synced scripts remain executable while honoring local security settings.

## How it works

1. **Resolve version** — `git ls-remote` converts the version to a commit SHA.
2. **Check lock** — Skip if `b.lock` already has the same commit.
3. **Clone/fetch** — Bare clone to `~/.cache/b/repos/`, fetch the target commit.
4. **Match files** — `git ls-tree` + glob matching, filtering ignored paths.
5. **Detect changes** — Compare on-disk SHA256 against `b.lock` checksums.
6. **Apply strategy** — Write files per the configured strategy.
7. **Update lock** — Record commit SHA, file checksums, modes, and destinations.

## Lock file

`b.lock` tracks the exact state of every synced env:

```json
{
  "envs": [
    {
      "ref": "github.com/org/infra",
      "label": "",
      "version": "v2.0",
      "commit": "abc123def456...",
      "previousCommit": "old789...",
      "files": [
        {
          "path": "manifests/base/deploy.yaml",
          "dest": "base/deploy.yaml",
          "sha256": "e3b0c44298fc...",
          "mode": "644"
        }
      ]
    }
  ]
}
```

Commit `b.lock` to version control so your team stays in sync and `b verify` can detect drift.

## Git cache

```bash
b cache path              # Show cache location
du -sh $(b cache path)    # Check cache size
b cache clean             # Remove all cached repos
```

Uses shallow bare clones (`--depth 1`) to minimize disk usage.

## Tips

- **CI/CD**: `replace` applies silently in non-interactive mode. Use `client` or `merge` if your pipeline modifies synced files.
- **Verify**: `b verify` checks all artifacts against `b.lock` checksums.
- **Conflicts**: If two env entries write to the same path, **b** warns before syncing.
- **Auth**: Private repos need `GITHUB_TOKEN`, `GITLAB_TOKEN`, or `GITEA_TOKEN`. See [Authentication](/authentication).
- **Force**: `b update --force` re-syncs even when up-to-date.
- **Unchanged**: Files identical to upstream are automatically skipped.
